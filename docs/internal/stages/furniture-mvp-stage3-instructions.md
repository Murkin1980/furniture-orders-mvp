# Furniture Orders MVP — инструкции для Этапа 3

Этот файл описывает Этап 3 проекта `furniture-orders-mvp`: переход от простой операционной панели со статусами к управляемому мебельному проекту с шаблонами, этапами и чек-листами выполнения. На этом этапе заказ перестаёт быть только строкой в таблице и превращается в рабочую единицу, по которой менеджер, конструктор и производство могут двигаться по стандартному сценарию без потери деталей.

## Цель Этапа 3

После Этапа 2 система уже умеет принимать заявки, показывать список заказов и менять их статус. Следующий шаг — стандартизировать исполнение: для каждого заказа нужен понятный проектный шаблон, список обязательных шагов и базовая структура внутренних этапов. Практика custom furniture workflows обычно включает зафиксированные стадии вроде brief, design, approval, production, finishing, delivery and installation, а также контроль спецификаций, материалов, сроков и quality checks. [web:160][web:162][web:163][web:166][web:169]

## Что должно получиться в конце этапа

После завершения Этапа 3 проект должен уметь:

- создавать заказу шаблон исполнения;
- хранить набор шагов или чек-листов по заказу;
- отмечать выполнение шагов;
- показывать проектную структуру заказа в admin-интерфейсе;
- использовать разные шаблоны для разных типов мебели;
- не ломать уже существующие API Этапов 1 и 2. [web:158][web:160][web:162][web:169]

## Основная идея этапа

На этом этапе не нужно строить полноценную ERP. Достаточно ввести лёгкую project layer над заказом:

- `order` остаётся бизнес-сущностью для лида и сделки;
- `project template` задаёт типовой путь выполнения;
- `project steps` или `checklist items` фиксируют конкретные действия;
- менеджер видит, что по заказу уже сделано, а что ещё нет.

Для мебельного бизнеса это особенно полезно, потому что custom orders часто зависят от точных размеров, подтверждённых материалов, согласованных чертежей, закупки фурнитуры, производства, доставки и монтажа. Ошибки обычно происходят не из-за отсутствия статуса, а из-за отсутствия стандартного набора обязательных шагов. [web:162][web:163][web:164][web:169]

## Базовая модель данных для Этапа 3

Рекомендуется не перегружать существующую БД, а добавить три новых слоя.

### 1. Шаблоны проектов

Новая таблица:

```text
project_templates
```

Минимальные поля:

- `id`
- `code` — короткий системный код шаблона, например `kitchen-basic`
- `name` — человекочитаемое имя
- `furniture_type` — например `kitchen`, `wardrobe`, `table`, `sofa`
- `is_active`
- `created_at`

### 2. Шаги шаблона

Новая таблица:

```text
template_steps
```

Минимальные поля:

- `id`
- `template_id`
- `step_code`
- `title`
- `sort_order`
- `required`
- `default_assignee_role` — например `manager`, `designer`, `production`

### 3. Шаги заказа

Новая таблица:

```text
order_steps
```

Минимальные поля:

- `id`
- `order_id`
- `template_step_id`
- `step_code`
- `title`
- `status` — `pending`, `done`, `skipped`
- `notes`
- `completed_at`
- `completed_by`
- `sort_order`

Такой подход позволит один раз описать шаблон и создавать для каждого нового заказа копию шагов в виде рабочего чек-листа.

## Минимальный SQL-план

Новая миграция может выглядеть так:

```sql
CREATE TABLE IF NOT EXISTS project_templates (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code TEXT NOT NULL UNIQUE,
  name TEXT NOT NULL,
  furniture_type TEXT NOT NULL,
  is_active INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS template_steps (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  template_id INTEGER NOT NULL,
  step_code TEXT NOT NULL,
  title TEXT NOT NULL,
  sort_order INTEGER NOT NULL DEFAULT 0,
  required INTEGER NOT NULL DEFAULT 1,
  default_assignee_role TEXT,
  FOREIGN KEY (template_id) REFERENCES project_templates(id) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS order_steps (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id INTEGER NOT NULL,
  template_step_id INTEGER,
  step_code TEXT NOT NULL,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  notes TEXT,
  completed_at TEXT,
  completed_by TEXT,
  sort_order INTEGER NOT NULL DEFAULT 0,
  FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE,
  FOREIGN KEY (template_step_id) REFERENCES template_steps(id) ON DELETE SET NULL
);
```

Также стоит добавить индексы:

```sql
CREATE INDEX IF NOT EXISTS idx_template_steps_template_id ON template_steps(template_id);
CREATE INDEX IF NOT EXISTS idx_order_steps_order_id ON order_steps(order_id);
CREATE INDEX IF NOT EXISTS idx_order_steps_status ON order_steps(status);
```

## Базовые шаблоны для MVP

На старте не нужно создавать десятки шаблонов. Достаточно трёх:

### 1. Kitchen

Пример шагов:

1. Уточнить размеры помещения.
2. Подтвердить планировку.
3. Согласовать материалы и фасады.
4. Согласовать фурнитуру.
5. Подтвердить бюджет.
6. Подготовить финальный расчёт.
7. Подписать/подтвердить заказ.
8. Запустить в производство.
9. Контроль готовности.
10. Доставка и монтаж.
11. Финальная сдача.

Такие проверки соответствуют типовым требованиям к подтверждению размеров, материалов, finish codes, revision control и site readiness. [web:162][web:166][web:169]

### 2. Wardrobe / шкаф

Пример шагов:

1. Замер ниши или помещения.
2. Подтверждение конфигурации.
3. Подтверждение наполнения.
4. Согласование цвета и материала.
5. Согласование стоимости.
6. Производство.
7. Доставка.
8. Монтаж.
9. Приёмка.

### 3. Table / простой корпусный заказ

Пример шагов:

1. Уточнить размеры.
2. Согласовать форму и материал.
3. Подтвердить отделку.
4. Согласовать цену.
5. Запуск в производство.
6. Контроль качества.
7. Выдача или доставка.

## Архитектурный принцип

Шаблон не должен жёстко управлять статусом заказа. На Этапе 3 лучше оставить две параллельные оси:

- `order.status` — крупный бизнес-статус (`new`, `in_review`, `quoted`, `in_production`, `completed`, `canceled`);
- `order_steps` — детальная операционная работа внутри заказа.

Это важно, потому что один и тот же статус `in_review` может включать несколько внутренних действий: замер, согласование материалов, подтверждение бюджета и правки проекта. [web:162][web:163][web:169]

## Что нужно реализовать в коде

### Шаг 1. Создать модуль шаблонов

Например:

```text
src/project-templates.js
```

Внутри:

- сидовые шаблоны для MVP;
- функция поиска шаблона по `furnitureType`;
- функция получения шагов шаблона.

### Шаг 2. Создать функции работы с чек-листом

В `src/orders-core.js` или в новом модуле добавить функции:

- `createOrderProjectSteps({ db, orderId, furnitureType })`
- `listOrderSteps({ db, orderId })`
- `updateOrderStep({ db, orderId, stepId, status, notes, completedBy })`

Что они должны делать:

- при создании проектной структуры копировать шаги шаблона в `order_steps`;
- читать шаги заказа в отсортированном порядке;
- обновлять отдельный шаг;
- не позволять задавать неизвестный `status` шага.

### Шаг 3. Привязать генерацию шагов к заказу

Есть два допустимых варианта:

#### Вариант A — создавать шаги сразу при создании заказа

Плюсы:
- всё готово автоматически;
- админка сразу показывает проектные шаги.

Минусы:
- для мусорных или недозрелых лидов появятся лишние steps.

#### Вариант B — создавать шаги после перехода в `in_review`

Плюсы:
- project layer появляется только у реальных активных заказов;
- меньше шума в базе.

Минусы:
- появляется дополнительная точка инициализации.

Для вашего MVP я рекомендую **Вариант B**: создавать project steps, когда заказ впервые переводится в `in_review`. Это лучше соответствует реальной воронке мебельной студии, где не каждый новый лид сразу становится полноценным проектом.

## Новые API для Этапа 3

Минимальный набор endpoints:

### 1. Получить шаги заказа

```text
GET /api/orders/:id/steps
```

Если не хочется усложнять роутинг, допустим также вариант:

```text
GET /api/order-steps?orderId=123
```

Ответ:

```json
{
  "success": true,
  "items": [
    {
      "id": 1,
      "stepCode": "measurements_confirmed",
      "title": "Подтвердить размеры",
      "status": "pending",
      "notes": null,
      "completedAt": null,
      "completedBy": null,
      "sortOrder": 10
    }
  ]
}
```

### 2. Обновить шаг заказа

```text
POST /api/order-steps/update
```

Тело запроса:

```json
{
  "orderId": 12,
  "stepId": 5,
  "status": "done",
  "notes": "Замер подтверждён 30.05.2026",
  "completedBy": "manager"
}
```

### 3. Инициализировать шаги для заказа

Если выбран отложенный сценарий создания steps:

```text
POST /api/orders/project/init
```

Тело запроса:

```json
{
  "orderId": 12
}
```

Endpoint:
- определяет `furnitureType`;
- подбирает шаблон;
- создаёт `order_steps`, если их ещё нет;
- возвращает список шагов.

## Изменения в admin page

Admin page на Этапе 3 не нужно полностью переписывать. Достаточно расширить текущую админку.

Минимальный интерфейс:

- в таблице заказов кнопка `Открыть проект`;
- справа или ниже блок `Шаги заказа`;
- список шагов с чекбоксом или select;
- заметка к каждому шагу;
- индикатор прогресса, например `4/10 выполнено`.

Этого уже достаточно, чтобы менеджер видел не только статус заказа, но и фактическое состояние работы.

## Правила для шаблонов и шагов

Чтобы система не превратилась в хаос, надо заранее закрепить правила:

- у каждого шага должен быть `step_code`;
- у каждого шаблона должен быть стабильный `code`;
- `sort_order` определяет отображение;
- шаги не удаляются физически при выполнении;
- статусы шага ограничены списком `pending`, `done`, `skipped`;
- если шаг выполнен, ставится `completed_at`;
- если шаг возвращён в `pending`, `completed_at` можно очищать.

## Definition of Done для Этапа 3

Этап 3 считается завершённым, если выполнены все пункты:

- миграция для шаблонов и `order_steps` создана и применяется;
- есть минимум 3 рабочих шаблона;
- при инициализации заказа создаются шаги проекта;
- список шагов читается через API;
- отдельный шаг обновляется через API;
- admin page показывает шаги заказа;
- в UI можно отметить шаг как `done` или `skipped`;
- у шага можно сохранить заметку;
- Этапы 1 и 2 не сломаны (`POST /api/orders`, `GET /api/orders`, `POST /api/orders/status` продолжают работать);
- `npm run check` и `npm test` проходят.

## Рекомендуемый порядок реализации

Делать в таком порядке:

1. Миграция БД.
2. Seed шаблонов.
3. Функции ядра для работы с шагами.
4. API чтения шагов.
5. API обновления шага.
6. Ленивая инициализация project steps.
7. UI в admin page.
8. Автотесты и ручные проверки.

Это удержит проект в рабочем состоянии на каждом промежуточном шаге.

## Что не делать на Этапе 3

Пока не нужно:

- строить сложный календарь производства;
- делать многопользовательские роли и ACL;
- вводить отдельные доски Kanban по цехам;
- делать генерацию BOM;
- подключать складской учёт;
- добавлять полный документооборот договоров;
- синхронизировать всё это с CRM, ERP и бухгалтерией.

Этап 3 должен остаться лёгким: шаблоны, шаги, отметка выполнения, заметки и минимальный прогресс проекта. [web:164][web:165]

## Что даст Этап 3 бизнесу

Практический эффект Этапа 3:

- меньше забытых шагов по заказу;
- меньше устных договорённостей без фиксации;
- проще передавать заказ от менеджера к производству;
- проще обучать нового администратора;
- легче в будущем автоматизировать напоминания, уведомления и AI-подсказки.

Именно здесь MVP начинает превращаться в универсальный операционный шаблон для мебельной студии, а не просто в форму заявок. [web:160][web:162][web:164][web:169]

## Короткий вывод

Этап 3 — это слой стандартизации исполнения. Если Этап 1 собирал лиды, а Этап 2 дал управление статусами, то Этап 3 даёт проектную дисциплину: что именно надо сделать по заказу, в каком порядке и что уже выполнено. [web:160][web:162][web:163][web:169]
