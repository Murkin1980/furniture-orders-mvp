# Furniture Orders MVP — инструкции для Этапа 2

Этот файл описывает Этап 2 проекта: переход от простого приёма заявок к минимальной операционной системе для мебельной компании. После завершения этапа менеджер должен видеть список заказов, менять статусы и работать с базовой админкой без разрушения уже готового intake-потока из Этапа 1.

## Цель Этапа 2

Этап 1 уже дал фундамент: форма принимает заявку, backend валидирует payload, D1 сохраняет клиента и заказ, Telegram уведомляет менеджера. Теперь задача — добавить минимальный operational layer: очередь заказов, статусы, просмотр и обновление заявки. Такая логика соответствует типовым CRM и order-tracking сценариям в мебельном бизнесе, где важно отслеживать движение заказа от продажи к производству и завершению. [web:140][web:147][web:153][web:154]

## Что должно получиться в конце этапа

После завершения Этапа 2 проект должен уметь:

- хранить не только новые заявки, но и изменения по ним;
- отдавать список заказов менеджеру;
- фильтровать заказы по статусу;
- менять статус заказа через API;
- показывать простой admin view со списком заказов;
- сохранять обратную совместимость с Этапом 1, чтобы входящий контракт заявки не сломался. [web:140][web:141][web:153]

## Статусы для MVP

Для второго этапа не нужно строить сложный жизненный цикл производства. Достаточно короткого и понятного набора статусов:

- `new` — новая заявка;
- `in_review` — менеджер уточняет задачу;
- `quoted` — клиенту отправлен просчёт или предложение;
- `in_production` — заказ пошёл в работу;
- `completed` — заказ выполнен;
- `canceled` — заказ отменён.

Для MVP этого достаточно, чтобы видеть рабочий поток и не перегружать интерфейс. Практика custom statuses в order workflows как раз используется для понятной передачи заказа между этапами работы. [web:144][web:153][web:154]

## Этап 2 — план работ

### Шаг 1. Расширить схему базы

Нужно подготовить новую миграцию, которая добавит минимум:

- `updated_at` в таблицу `orders`;
- `notes` для внутренних комментариев менеджера.

Пример миграции:

```sql
ALTER TABLE orders ADD COLUMN updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP;
ALTER TABLE orders ADD COLUMN notes TEXT;
```

Если SQLite/D1 не позволит безопасно добавить столбец с `NOT NULL DEFAULT CURRENT_TIMESTAMP` в уже существующую таблицу в нужной форме, можно использовать более аккуратную стратегию миграции: добавить nullable-столбец, затем заполнить его из `created_at`, а уже потом использовать в коде как обязательный. Это лучше, чем ломать существующую таблицу. [file:136]

### Шаг 2. Зафиксировать допустимые статусы в коде

В `src/orders-core.js` или в отдельном модуле нужно ввести список допустимых статусов.

Пример:

```js
export const ORDER_STATUSES = [
  "new",
  "in_review",
  "quoted",
  "in_production",
  "completed",
  "canceled"
];
```

Это даст единый источник правды для backend и будущей admin UI.

### Шаг 3. Добавить чтение списка заказов

Нужен новый endpoint:

- `GET /api/orders`

Минимальные требования:

- возвращает список заказов;
- сортирует по `created_at DESC`;
- умеет фильтровать по `status` через query string;
- по возможности возвращает имя и телефон клиента вместе с заказом.

Пример ответа:

```json
{
  "success": true,
  "items": [
    {
      "id": 12,
      "clientId": 5,
      "clientName": "Ерлан",
      "phone": "+77011234567",
      "source": "site",
      "city": "Алматы",
      "furnitureType": "kitchen",
      "budget": 850000,
      "status": "new",
      "createdAt": "2026-05-29 19:00:00",
      "updatedAt": "2026-05-29 19:00:00"
    }
  ]
}
```

Для MVP этого уже хватает, чтобы менеджер видел живую очередь заказов. [web:140][web:153]

### Шаг 4. Добавить обновление статуса

Нужен endpoint для смены статуса. Можно пойти одним из двух путей:

#### Вариант A — REST-путь с id

```text
PATCH /api/orders/:id
```

Тело запроса:

```json
{
  "status": "in_review",
  "notes": "Клиенту нужно уточнить размеры"
}
```

#### Вариант B — отдельный action endpoint

```text
POST /api/orders/status
```

Тело запроса:

```json
{
  "orderId": 12,
  "status": "in_review",
  "notes": "Клиенту нужно уточнить размеры"
}
```

Для текущего MVP я бы рекомендовал Вариант B, потому что он проще в реализации на текущей структуре проекта и не требует усложнять роутинг Pages Functions.

Минимальная логика endpoint:

- проверить наличие `orderId` и `status`;
- убедиться, что `status` входит в `ORDER_STATUSES`;
- обновить `status`, `updated_at`, `notes`;
- вернуть обновлённую запись или короткий success response. [web:144]

### Шаг 5. Подготовить функции чтения и обновления в ядре

Нужно добавить в `src/orders-core.js` две новые функции:

- `listOrders({ db, status })`
- `updateOrderStatus({ db, orderId, status, notes })`

Что должна делать `listOrders()`:

- читать заказы из таблицы `orders`;
- делать join с `clients`;
- поддерживать фильтр по статусу;
- возвращать чистый JSON без лишней внутренней структуры D1.

Что должна делать `updateOrderStatus()`:

- валидировать входные данные;
- проверять существование заказа;
- обновлять `status`, `notes`, `updated_at`;
- возвращать `404`, если заказ не найден;
- возвращать `400`, если статус недопустим.

### Шаг 6. Сделать минимальную admin page

Нужна простая страница `/admin` или `public/admin.html`.

На первом проходе достаточно такого интерфейса:

- таблица заказов;
- колонка клиента;
- тип мебели;
- город;
- бюджет;
- текущий статус;
- селект или кнопки для смены статуса;
- поле заметки;
- кнопка обновить.

Важно: не превращать это в сложную SPA. Для Этапа 2 достаточно очень простой HTML-страницы с `fetch()` к `GET /api/orders` и `POST /api/orders/status`.

### Шаг 7. Минимальная защита admin page

На этом этапе не нужно строить полноценную auth-систему. Для MVP достаточно одной простой защиты:

- basic token через env и заголовок;
- или простой пароль на странице и проверка через endpoint.

Главное — не оставлять `/admin` полностью открытым, если сайт доступен в интернете.

### Шаг 8. Подготовить ручные тесты

Нужно протестировать:

1. Новый заказ попадает в список.
2. Фильтр по `status=new` работает.
3. Статус заказа меняется на `in_review`.
4. `updated_at` обновляется.
5. При неверном статусе API возвращает `400`.
6. При несуществующем заказе API возвращает `404`.
7. Intake flow из Этапа 1 не ломается после всех изменений.

## Рекомендуемая структура файлов

После Этапа 2 проект может выглядеть так:


Если не хочется плодить много файлов, можно оставить меньше файлов, но логически разделение чтения списка и обновления статуса всё равно полезно.

## Definition of Done для Этапа 2

Этап 2 считается завершённым, если выполнены все пункты:

- существует миграция для `updated_at` и `notes`;
- в коде есть единый список допустимых статусов;
- работает `GET /api/orders`;
- работает фильтр по статусу;
- работает endpoint смены статуса;
- `updated_at` реально меняется при апдейте;
- admin page показывает список заказов;
- admin page позволяет сменить статус хотя бы у одного заказа;
- старый путь `POST /api/orders` продолжает работать без изменений;
- `npm run check` и `npm test` проходят после всех изменений. [file:133]

## Практический порядок реализации

Делать именно в таком порядке:

1. Миграция БД.
2. Константа статусов.
3. `listOrders()` в ядре.
4. Endpoint списка заказов.
5. `updateOrderStatus()` в ядре.
6. Endpoint обновления статуса.
7. Простая admin page.
8. Ручные и автоматические тесты.

Это важно: сначала backend и данные, потом UI. Иначе очень легко уйти в интерфейс без стабильного API.

## Что не делать на Этапе 2

Пока не нужно:

- делать полноценную CRM;
- добавлять сложную RBAC-авторизацию;
- строить real-time обновления;
- подключать AI-агента;
- вводить шаблоны производственных задач;
- делать мультиарендность.

Этап 2 должен остаться лёгким и понятным: менеджер видит заказы и двигает их по статусам. [web:141][web:153]
