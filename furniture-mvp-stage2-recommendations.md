# Furniture Orders MVP — рекомендации по Этапу 2

Этот файл фиксирует итоговые рекомендации после реализации Этапа 2 проекта `furniture-orders-mvp`. Его задача — не описывать заново весь этап, а собрать в одном месте все выводы, риски, доработки и post-stage действия перед переходом к Этапу 3.

## Итоговая оценка Этапа 2

Этап 2 выполнен хорошо: проект действительно перешёл от простого intake flow к минимальной операционной панели для мебельной компании. По итоговому summary реализованы список заказов, фильтрация по статусу, изменение статуса, внутренние заметки, admin token, admin page, обновлённый README, автоматические тесты и production deploy. Это уже полноценный operational layer поверх intake-потока Этапа 1. [file:156]

Особенно важно, что старый endpoint `POST /api/orders` остался совместимым с предыдущим этапом, а новые управленческие функции были добавлены поверх уже существующего ядра. Это правильный путь для универсального решения под мебельные компании: расширять систему слоями, а не ломать работающий входящий поток заявок. [file:156]

## Что в Этапе 2 сделано правильно

### 1. Статусы вынесены в отдельный источник правды

Выделение списка статусов в `src/order-statuses.js` и добавление `isOrderStatus(value)` — очень правильное решение. Это снижает риск рассинхронизации между backend, UI и тестами, а также упрощает развитие следующих этапов. [file:156]

### 2. Добавлены операционные endpoints

Появление `GET /api/orders` и `POST /api/orders/status` делает систему уже не просто формой заявок, а рабочим инструментом для менеджера. Это соответствует задаче мебельного order tracking: видеть очередь заказов, фильтровать и двигать их по статусам. [file:156]

### 3. Admin page сделана без лишней сложности

Использование `public/admin.html` на ванильном HTML/CSS/JS — правильный выбор для текущего уровня зрелости продукта. Для MVP это лучше, чем преждевременно тянуть SPA-фреймворк и усложнять деплой, состояние и сопровождение. [file:156]

### 4. Есть защита admin API

Добавление `ADMIN_TOKEN` — хороший компромисс между безопасностью и простотой. Для раннего MVP этого достаточно, особенно при поддержке двух способов передачи токена через `X-Admin-Token` и `Authorization: Bearer`. [file:156]

### 5. Есть тесты и ручные проверки

Наличие 11 passing tests, ручных API-проверок и визуальной проверки admin UI сильно повышает доверие к этапу. Это уже не “концепт”, а подтверждённая рабочая версия. [file:156]

## Главные рекомендации после завершения Этапа 2

### 1. Перевыпустить production `ADMIN_TOKEN`

В summary зафиксировано, что локальная копия production token была сохранена в файле:

```text
C:\tmp\furniture-orders-admin-token.txt
```

Это не означает утечку автоматически, но с точки зрения operational hygiene такой токен лучше считать потенциально засвеченным. После этого этапа рекомендуется:

- сгенерировать новый `ADMIN_TOKEN`;
- обновить secret в Cloudflare Pages;
- удалить старый локальный файл;
- не хранить production токен в обычных временных файлах без отдельной дисциплины доступа. [file:156]

### 2. Уточнить историю миграций по `updated_at`

В summary указано, что новая миграция добавила `notes`, а `updated_at` уже был в системе ранее. Это нормально, если так и есть, но в репозитории и README должно быть абсолютно понятно:

- в какой миграции появился `updated_at`;
- входит ли он в базовую миграцию для новых баз;
- нужно ли что-то делать отдельно при поднятии базы с нуля.

Иначе следующий разработчик может запутаться в реальном состоянии схемы. [file:156]

### 3. Упростить и формализовать migration story

Сейчас проект уже перешёл на стадию, где миграции начинают иметь значение не только для локальной проверки, но и для стабильного развёртывания. Рекомендуется:

- убедиться, что все миграции воспроизводимы на чистой базе;
- проверить, что новая база поднимается без ручных правок SQL;
- описать это в README отдельным коротким блоком “deploy from scratch”. [file:156]

### 4. Отделить рабочие документы от кода ещё аккуратнее

В summary уже указано, что markdown-файлы этапов оставлены untracked, а `output/` добавлен в `.gitignore`. Это хорошо. Следующий шаг — договориться, где будут храниться такие документы постоянно:

- либо в отдельной папке `docs/` и коммитить их осознанно;
- либо всегда держать вне репозитория как временные рабочие файлы.

Главное — не допустить ситуации, когда часть архитектурных решений живёт только в чате, а часть — в случайных untracked markdown-файлах. [file:156]

### 5. Проверить использование `localStorage` в admin page

В admin page токен сохраняется в `localStorage`. Для обычного браузерного сценария это допустимо, и для MVP такая реализация практична. Но в долгосрочной перспективе это стоит считать техническим долгом, потому что:

- токен живёт в браузере дольше, чем нужно;
- при доступе с общего компьютера это может быть неудобно;
- позже, при росте требований к безопасности, лучше перейти на более контролируемую схему хранения. [file:156]

Пока это не блокер, но это стоит отметить как задачу на будущее.

### 6. Добавить smoke test на admin auth

В summary уже есть ручные проверки `401 unauthorized` и `503 admin_not_configured`. Это хорошо, но полезно закрепить хотя бы один автоматический smoke test для admin authorization:

- запрос без токена → `401`;
- запрос с неверным токеном → `401`;
- запрос с валидным токеном → `200`.

Это дешёвая, но полезная защита от регрессий. [file:156]

### 7. Стабилизировать admin API как публичный внутренний контракт

Теперь у проекта уже есть не только внешний intake API, но и внутренний operational API. Значит, есть смысл зафиксировать его как мини-контракт:

- формат ответа `GET /api/orders`;
- формат ответа `POST /api/orders/status`;
- коды ошибок `400`, `401`, `404`, `503`;
- формат передачи токена.

Это пригодится на Этапе 3, когда начнут появляться шаблоны проектов, дополнительные действия по заказу и, возможно, AI-инструменты поверх admin API. [file:156]

## Что проверить вручную после завершения Этапа 2

Сделать короткий post-stage smoke checklist:

1. `POST /api/orders` создаёт новый заказ.
2. Новый заказ появляется в `/admin`.
3. Фильтр по `new` работает.
4. Смена статуса на `in_review` работает.
5. `notes` сохраняется и повторно читается.
6. `updated_at` реально изменяется после update.
7. Без токена admin API недоступен.
8. После production deploy всё это работает на боевом URL. [file:156]

Для текущего MVP это будет преждевременное усложнение. Система уже решает основную задачу: менеджер видит и двигает заказы. [file:156]
